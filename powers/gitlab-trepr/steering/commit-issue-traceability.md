# Commit-Issue Traceability Rule

## Policy

Every commit in this workspace MUST reference a GitLab issue. This ensures full traceability between code changes and the rationale (feature, bug, task) that motivated them.

## Workflow

When performing a commit, follow this sequence:

### 1. Identify the Related Issue

Before composing the commit message, determine if there is an existing GitLab issue that this commit addresses:

1. **Search for assigned issues automatically:**
   - Resolve the current project path (see "Project Path Resolution" below)
   - Use the `list_issues` tool to fetch open issues assigned to the current user in the project
   - Compare issue titles and descriptions against the work being committed (files changed, commit subject, branch name)
   - If a matching issue is found, present it to the user for confirmation: "I found issue #N (`<title>`) assigned to you that seems related to this change. Should I reference it in the commit?"

2. **If the user confirms the suggested issue**, use that issue IID

3. **If no matching assigned issue is found**, or the user rejects the suggestion:
   - Ask the user: "Which GitLab issue does this commit relate to?"
   - If the user provides an issue number (e.g., `#42`), use it
   - If the user provides a description but no issue number, search all open issues in the project for a match

### 2. Create Issue If None Exists

If no existing issue matches the work being committed:

1. **Activate the `gitlab-trepr` power** (use `kiro_powers action="activate" powerName="gitlab-trepr"`)
2. **Check `GITLAB_READ_ONLY_MODE`** — if `true`, inform the user that an issue cannot be created automatically and ask them to create one manually or change the setting
3. **Create the issue** using the `create_issue` tool with:
   - A title that summarizes the work (in Portuguese, matching commit subject conventions)
   - A description explaining the context
   - Appropriate labels if identifiable from the commit type (e.g., `bug` for `fix`, `enhancement` for `feat`)
4. **Note the issue IID** returned by the API for use in the commit footer

### 3. Compose Commit Message with Issue Reference

The commit message footer MUST include a reference to the GitLab issue using one of these patterns:

```
Closes: #<IID>        # When the commit fully resolves the issue
References: #<IID>    # When the commit partially addresses or relates to the issue
```

These patterns follow the Conventional Commits footer convention already defined in the `git-operator` skill.

### 4. Mapping Commit Types to Issue Labels

When creating an issue automatically, use this mapping:

| Commit type | Suggested label |
|-------------|----------------|
| `feat`      | `enhancement`  |
| `fix`       | `bug`          |
| `docs`      | `documentation`|
| `chore`     | `maintenance`  |
| `test`      | `test`         |
| `style`     | `style`        |

## Examples

### Commit with existing issue:
```
feat(auth): implementar autenticação OAuth2 no login

- Adicionado fluxo de Authorization Code
- Integração com provedor interno do GitLab

Closes: #78
Assisted-by: Kiro
```

### Commit where issue is created first:
```
fix(api): corrigir timeout na listagem de projetos

- Aumentado timeout de conexão para 30s
- Adicionado retry com backoff exponencial

Closes: #142
Assisted-by: Kiro
```

## Project Path Resolution

When creating issues, the agent must determine the correct GitLab project path:

1. Check `git remote get-url origin` to extract the project path from the remote URL
2. Convert the SSH/HTTPS URL to the namespace format (e.g., `git@gitlab.tre-pr.jus.br:sds/sistemas/meu-projeto.git` → `sds/sistemas/meu-projeto`)
3. If the remote URL cannot be parsed, ask the user for the full project path

## Exceptions

The following commits MAY be exempt from the issue requirement (user must explicitly confirm the exemption):

- Initial commit in a new repository
- Merge commits (generated by Git, not authored manually)
- Commits that only modify `.gitignore`, `.gitattributes`, or similar meta-files with no functional impact
